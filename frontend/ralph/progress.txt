## 2026-04-19 - US-012 - Reduced-motion + scroll-progress + mounted hooks
- Authored three client hooks under `src/hooks/`:
  - `use-reduced-motion.ts` ‚Äî reads `window.matchMedia('(prefers-reduced-motion: reduce)')` inside `useEffect`. Initial state is `false` so SSR stays stable; the client reconciles after the first effect. Subscribes with `addEventListener('change', handler)` + cleans up with `removeEventListener` on unmount. Guards `typeof window` and `typeof window.matchMedia` for exotic SSR/test environments.
  - `use-mounted.ts` ‚Äî flips `false ‚Üí true` inside a zero-dep `useEffect`. Minimal but invaluable for gating client-only UI (e.g. the US-013 Three.js canvas) so server markup stays stable.
  - `use-scroll-progress.ts` ‚Äî accepts a `RefObject<HTMLElement | null>` and returns 0‚Äì1 progress across `(viewport + element height)`. Registers passive scroll + resize listeners, recomputes with `getBoundingClientRect()`, and clamps with a local `clamp01()` helper. Cleans up listeners on unmount. `noUncheckedIndexedAccess` means `ref.current` is typed `HTMLElement | null`, so we early-return on null.

- **Learnings for future iterations:**
  - `renderHook` from `@testing-library/react@^16` flushes effects inside an implicit `act()` wrapper ‚Äî `result.current` is the POST-mount value, not the initial render. If you ever need to assert the initial/pre-effect state, you'd have to roll a custom renderer; for this codebase's hooks we only care about the post-mount behavior, which is what downstream components see.
  - jsdom's built-in `matchMedia` stub returns `{ matches: false, addEventListener: () => {} }` by default ‚Äî meaning change events never fire. Any test asserting on media-query updates MUST override `window.matchMedia` via `Object.defineProperty(window, 'matchMedia', { configurable: true, writable: true, value: vi.fn()... })`. Use `afterEach(() => vi.restoreAllMocks())` to keep tests isolated ‚Äî `vi.fn()` mocks are restored but `defineProperty` assignments persist; since we rewrite `window.matchMedia` at the top of each test, this is fine in practice.
  - `RefObject<T | null>` vs `RefObject<T>` matters: with `noUncheckedIndexedAccess` + React 19 typings, `useRef<HTMLElement>(null)` actually produces `RefObject<HTMLElement | null>`. So `useScrollProgress(ref)` must accept the nullable form ‚Äî if the acceptance-criterion reads as just "ref", the safest API surface is `RefObject<HTMLElement | null>` and an internal null guard. Do NOT `ref.current!` ‚Äî explicit early-return is clearer.
  - The scroll-progress formula `(viewport - rect.top) / (rect.height + viewport)` hits 0 when the element's top is just entering from below the viewport bottom and 1 when its bottom has just exited the top. Two common alternatives: `(viewport - rect.top) / viewport` (0‚Üí1 as element passes viewport, used by some parallax libs ‚Äî ignores element height) and `-rect.top / rect.height` (enters at top, exits at bottom ‚Äî only maps between 0 and 1 while pinned). The chosen formula is the most symmetric and matches what GSAP ScrollTrigger's `scrub` progress gives you between `top bottom` and `bottom top` triggers ‚Äî so US-014/US-015 can use the hook OR ScrollTrigger progress interchangeably.
  - `useScrollProgress` uses `{ passive: true }` on `scroll` specifically to keep the compositor thread happy ‚Äî React 19's concurrent renderer + a non-passive scroll listener can produce 20‚Äì30ms jank per frame. If you ever need `event.preventDefault()` in a scroll handler (you don't here), use a separate non-passive listener for that purpose.
  - Every hook file starts with `"use client"` (double quotes ‚Äî matches Next.js docs and the house style). When a server component imports a client hook, Next's bundler splits at the boundary; the server component stays server-rendered. No need to mark every page `"use client"`.
  - Never ship a React hook without a named export from a `use-<name>.ts` file. `export default` breaks refactor tooling and mangles import names across the codebase; named exports make grep-for-usage trivial.
