- With Three.js `ShaderMaterial`, strict TS + `noUncheckedIndexedAccess` makes `material.uniforms.uX.value` mutation noisy because the uniform record type is `{ [k: string]: IUniform | undefined }`. Best practice: hold direct `const uProgress = { value: 0 }` references outside the uniforms dict and pass them INTO `uniforms: { uProgress, ... }` — Three sees the same object, and strict TS sees a non-nullable field. Cleaner than per-access `!` assertions in the rAF loop.
- For decorative Three.js particle fields, use `OrthographicCamera` + a `Points` mesh in normalized [-aspect, aspect] × [-1, 1] space. Simpler math than perspective, no frustum-culling surprises (also set `points.frustumCulled = false` — bounding box maths on a 12k vertex attribute is a per-frame waste when you know the whole field is in view). Pass `position` attribute as a placeholder (Three still reads it for bbox) even when the vertex shader ignores it.
- FPS-driven particle downgrade pattern: preallocate MAX_PARTICLES once, then scale render count via `geometry.setDrawRange(0, tierCount)` — no geometry rebuild, no GPU upload, tier switch is free. Track frames in a rolling 2s window; if fps < threshold AND tierIndex < TIERS.length-1, bump tier. Tiers 12000 → 6000 → 3000 gives ~4× headroom for low-power GPUs.
- rAF scroll-sync pattern: compute scroll progress into a mutable `let targetProgress` inside the effect, then critically-damped interpolate inside the rAF tick via `1 - Math.exp(-dt * k)` where `dt` is seconds since last frame. This decouples scroll event rate (up to 120 Hz on trackpads) from render rate (60 Hz) and survives scroll spikes without rubber-banding. Do NOT use `useScrollProgress()` for this — that hook's `setState` causes React re-renders for each scroll tick, which is exactly what you want to avoid inside a Three.js canvas.
- Three.js renderer cleanup on unmount is three calls in order: `cancelAnimationFrame(rafId)` → remove DOM listeners → `scene.remove(mesh)` → `geometry.dispose()` → `material.dispose()` → `renderer.dispose()` → `container.removeChild(renderer.domElement)`. React StrictMode's double-invoke in dev means this cleanup runs twice per mount — idempotency matters (use `if (parent === container)` guard on the removeChild call).
- `useReducedMotion()` short-circuits a Three scene to a single `renderer.render(scene, camera)` call AFTER setting the final uniforms (uProgress = current scroll, uTime = 0, no drift). No rAF loop is started, so there's zero ongoing CPU/GPU cost for reduced-motion users. The CSS `prefers-reduced-motion` rule in globals.css does NOT cover rAF loops — per-component JS short-circuit is mandatory.
- Mobile fallback pattern for WebGL: check `window.innerWidth < 768` on mount + resize; below the breakpoint, return a static SVG placeholder (ring + whatever glyph matches) instead of mounting the Three scene at all. This saves the full 120 kB Three chunk from even loading on phones — next/dynamic's SSR: false means the import resolves lazily, and if the effect never runs (isMobile branch), the chunk stays in the network queue only until the user crosses 768px.
- GLSL shader strings live in `src/components/landing/*/veil-shaders.ts` (or similar) as tagged `/* glsl */` template literals — the comment is a hint for editor highlighting, not parsed by TypeScript. Export VERTEX_SHADER + FRAGMENT_SHADER as const strings so ShaderMaterial can reference them directly. Use attribute names prefixed with `a` (aStart, aTarget) and uniform names prefixed with `u` (uProgress, uTime) to match the Three.js community convention — makes shader grep-ability trivial.
- Particle-field glyph encoding trick: split 12k particles between "glyph" (on the target silhouette) and "ambient" (drift-only) roles by encoding membership in the SIGN of the `aSize` attribute — negative = ambient, positive = glyph. The fragment shader reads `step(0.0, aSize)` into a `vGlyph` varying (0 or 1) and uses it to gate color shift + alpha fade per particle. Saves one attribute slot vs carrying a dedicated `aRole` attribute.
- GSAP registration lives ONCE in `src/lib/gsap.ts` (`"use client"` pragma at top) — idempotent `ensureRegistered()` guard + `typeof window !== "undefined"` check so `gsap.registerPlugin(ScrollTrigger)` never runs during SSR. Never `gsap.registerPlugin` in component files — always import `gsap`/`ScrollTrigger` from `@/lib/gsap` so registration happens exactly once. Re-registering isn't harmful but multiplying the import surface is.
- Typed matchMedia helper `brandMatchMedia(setup)` in `@/lib/gsap` wraps `gsap.matchMedia` with three fixed breakpoints (isDesktop = `min-width:768px AND prefers-reduced-motion:no-preference`, isMobile = `max-width:767px`, isReduced = `prefers-reduced-motion:reduce`). The setup fn receives a typed `BreakpointConditions` object (booleans for all three, never undefined) plus the raw `gsap.Context` for `.add()` access. Reduced-motion users fall into isReduced (not isDesktop) — so any desktop-only pin/scrub code inside `if (conditions.isDesktop)` is automatically reduced-motion-safe without a second check.
- `PinnedSection` (`src/components/motion/pinned-section.tsx`) is the canonical pin/scrub primitive. Pattern: `const mm = brandMatchMedia((conditions) => { if (!conditions.isDesktop) return; const trigger = ScrollTrigger.create({trigger: node, start: "top top", end: () => `+=${window.innerHeight * pinDuration}`, pin: true, scrub: true, invalidateOnRefresh: true, onUpdate: (self) => { ... self.progress ... }}); return () => trigger?.kill(); })`; cleanup = `mm.kill()` (which runs the returned teardown AND disposes the matchMedia context). `invalidateOnRefresh: true` so ScrollTrigger re-queries `window.innerHeight * pinDuration` on viewport resize. The `end` value MUST be a function (not a string/number) for invalidateOnRefresh to actually re-read it.
- onProgress callback pattern for scroll-driven client components: hold the callback in a `useRef` and sync via a separate `useEffect([onProgress])`. Main effect's dep array stays `[pinDuration]` (or similar stable deps) so the ScrollTrigger isn't torn down + re-created every render when the parent passes a new handler each render. Without the ref indirection, any parent that doesn't memoize `onProgress` re-mounts the pin on every render — visible as scroll rubber-banding.
- Landing first-load JS budget check (US-032 target ≤ 220 kB gzipped): pre-US-014 was 117 kB. GSAP + ScrollTrigger added ~46 kB → 163 kB. Remaining headroom ~57 kB for US-016 CountUp (motion v12), US-017 FadeIn (motion v12 — already installed, no extra weight if tree-shaken). Shiki (US-020) is server-only so no client impact. - **Learnings for future iterations:**
  - `renderHook` from `@testing-library/react@^16` flushes effects inside an implicit `act()` wrapper — `result.current` is the POST-mount value, not the initial render. If you ever need to assert the initial/pre-effect state, you'd have to roll a custom renderer; for this codebase's hooks we only care about the post-mount behavior, which is what downstream components see.
  - jsdom's built-in `matchMedia` stub returns `{ matches: false, addEventListener: () => {} }` by default — meaning change events never fire. Any test asserting on media-query updates MUST override `window.matchMedia` via `Object.defineProperty(window, 'matchMedia', { configurable: true, writable: true, value: vi.fn()... })`. Use `afterEach(() => vi.restoreAllMocks())` to keep tests isolated — `vi.fn()` mocks are restored but `defineProperty` assignments persist; since we rewrite `window.matchMedia` at the top of each test, this is fine in practice.
  - `RefObject<T | null>` vs `RefObject<T>` matters: with `noUncheckedIndexedAccess` + React 19 typings, `useRef<HTMLElement>(null)` actually produces `RefObject<HTMLElement | null>`. So `useScrollProgress(ref)` must accept the nullable form — if the acceptance-criterion reads as just "ref", the safest API surface is `RefObject<HTMLElement | null>` and an internal null guard. Do NOT `ref.current!` — explicit early-return is clearer.
  - The scroll-progress formula `(viewport - rect.top) / (rect.height + viewport)` hits 0 when the element's top is just entering from below the viewport bottom and 1 when its bottom has just exited the top. Two common alternatives: `(viewport - rect.top) / viewport` (0→1 as element passes viewport, used by some parallax libs — ignores element height) and `-rect.top / rect.height` (enters at top, exits at bottom — only maps between 0 and 1 while pinned). The chosen formula is the most symmetric and matches what GSAP ScrollTrigger's `scrub` progress gives you between `top bottom` and `bottom top` triggers — so US-014/US-015 can use the hook OR ScrollTrigger progress interchangeably.
  - `useScrollProgress` uses `{ passive: true }` on `scroll` specifically to keep the compositor thread happy — React 19's concurrent renderer + a non-passive scroll listener can produce 20–30ms jank per frame. If you ever need `event.preventDefault()` in a scroll handler (you don't here), use a separate non-passive listener for that purpose.
  - Every hook file starts with `"use client"` (double quotes — matches Next.js docs and the house style). When a server component imports a client hook, Next's bundler splits at the boundary; the server component stays server-rendered. No need to mark every page `"use client"`.
  - Never ship a React hook without a named export from a `use-<name>.ts` file. `export default` breaks refactor tooling and mangles import names across the codebase; named exports make grep-for-usage trivial.

- **Learnings for future iterations:**
  - Next.js 15 has a stale-build gotcha: if you ever see `unhandledRejection [Error [PageNotFoundError]: Cannot find module for page: /_document]` during `pnpm build`, `rm -rf .next` and rebuild. Happened once after a `git stash` + `git stash pop` cycle — the build cache got confused and thought it was a Pages Router project. Clean builds reproducibly pass. Not a gsap or motion issue.
  - GSAP's `gsap.matchMedia` is the CORRECT abstraction for responsive + prefers-reduced-motion gating. Do NOT write `window.matchMedia('(max-width: 767px)').matches` in a scroll-driven component — gsap.matchMedia returns a Context whose teardown hook correctly disposes ScrollTriggers when the breakpoint flips (e.g. user resizes desktop → mobile). A bare `matchMedia` check happens once on mount and leaks triggers across breakpoint changes.
  - ScrollTrigger's `end` accepts `string | number | () => string | number`. For pin durations tied to viewport height, ALWAYS use the function form: `end: () => \`+=\${window.innerHeight * pinDuration}\``. Combined with `invalidateOnRefresh: true`, this makes the pin re-compute its duration when the viewport resizes. A static string like `end: "+=800"` won't adapt to viewport rotation on mobile/tablet.
  - `gsap.Context`'s `conditions` property from matchMedia can theoretically be `undefined` in the types (GSAP typings are permissive). Always coerce to booleans explicitly in the typed wrapper — `Boolean(context.conditions?.isDesktop)` — so downstream setup functions see strict booleans, not `boolean | undefined`. Saves a dozen `conditions.isDesktop === true` checks across call sites.
  - Typing a return of `gsap.matchMedia()` as `gsap.MatchMedia` works out of the box. Same for the `self: ScrollTrigger` param in `onUpdate` — the `@types/gsap` ships with the gsap package (no separate `@types/gsap` install needed). Strict mode + `noUncheckedIndexedAccess` cause no friction with the GSAP API as long as you don't do `ScrollTrigger.getAll()[0]` without a length check.
  - onProgress-ref pattern: the scroll-driven `setProgress(self.progress)` inside `onUpdate` causes React re-renders at up to 60 Hz during scroll. If your parent component ALSO re-renders often and passes a fresh `onProgress` prop each render, listing `onProgress` in the effect's deps would recreate the ScrollTrigger on every parent render → visible as scroll rubber-banding and pin flicker. The `useRef<typeof onProgress>(onProgress)` + separate sync effect indirection keeps the pin mounted across parent re-renders.
  - GSAP (~46 kB gz) belongs in the shared first-load chunk, NOT a per-section lazy chunk. Multiple sections (PinnedSection, US-015 Two Realities, US-017 FadeIn) will all import from `@/lib/gsap`, so bundling once + reusing is strictly better than N lazy chunks that each re-ship GSAP. Compare to Three.js (~120 kB) where lazy-loading a single section (US-013 hero) IS worth it. Rule of thumb: if a dep is shared across 2+ sections, stay in the shared chunk; if only one section uses it and it's >100 kB, lazy-load.
  - When a Server Component (demo-section) wraps its content inside a Client Component (PinnedSection), React serializes the children as RSC descriptors and the client component mounts + hydrates normally. No "use client" pragma needed on the parent. This is the preferred composition pattern — keep sections as Server Components and drop client behavior in via nested `"use client"` primitives.
  - Demo section chose `pinDuration={1.5}` arbitrarily — 1.5× viewport height gives a feel of "scroll a bit past the section, it stays glued to the top". For US-015 Two Realities (which animates value-by-value PII dissolve), a longer pinDuration (2.0 or 2.5) will be more appropriate so each character of the redaction has perceptible scroll distance. Tune per-section.
  - Reduced-motion scenarios fall through PinnedSection as a plain pass-through `<div>`. The progress indicator (when `showProgress`) still renders BUT shows "Scroll progress · 0%" forever (no ScrollTrigger → no state update). If that visual becomes distracting for reduced-motion users, either hide it with `motion-reduce:hidden` or gate the showProgress prop on `useReducedMotion()`. US-015 Two Realities won't show a progress bar so this is moot for now.
